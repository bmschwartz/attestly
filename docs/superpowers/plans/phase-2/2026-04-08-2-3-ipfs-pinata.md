# Sub-Plan 2-3: IPFS / Pinata Integration Implementation Plan

> **For agentic workers:** REQUIRED SUB-SKILL: Use superpowers:subagent-driven-development (recommended) or superpowers:executing-plans to implement this plan task-by-task. Steps use checkbox (`- [ ]`) syntax for tracking.

**Goal:** Implement Pinata IPFS client, deterministic JSON serialization (RFC 8785), and survey/response pinning functions. This enables immutable, content-addressed storage of survey definitions and response data on IPFS, with CIDs referenced on-chain for independent verification.

**Architecture:** The IPFS module lives at `src/lib/ipfs/` and provides: (1) a Pinata client wrapper for pinning and fetching, (2) RFC 8785 deterministic JSON serialization for CID reproducibility, (3) survey and response JSON schemas matching the IPFS content format spec, and (4) high-level `pinSurvey` and `pinResponse` functions that serialize data deterministically and pin to Pinata. All IPFS operations are server-side only (called from background job handlers, never from the client).

**Tech Stack:** Pinata SDK (`pinata`), TypeScript, Zod (schema validation)

**Spec reference:** `docs/superpowers/specs/2026-04-05-ipfs-storage-design.md` (JSON formats, Pinata integration, data flows, pin naming convention)

**Depends on:** Sub-Plan 2-2 (EIP-712 library -- for SurveyMessage/SurveyResponseMessage types and blinded ID)

### Task 2: Create deterministic JSON serialization

**Files:**
- Create: `src/lib/ipfs/deterministic-json.ts`

- [ ] **Step 1: Implement RFC 8785 JSON Canonicalization**

Create `src/lib/ipfs/deterministic-json.ts`:

This is the most critical module for CID reproducibility. Any party re-serializing the same survey/response data must produce byte-identical JSON to get the same IPFS CID. The RFC 8785 JSON Canonicalization Scheme (JCS) specifies:

1. **Sorted keys:** object keys are sorted lexicographically (Unicode code point order) at every nesting level
2. **No insignificant whitespace:** no spaces after colons or commas, no indentation, no trailing newlines
3. **UTF-8 encoding:** standard UTF-8, no BOM
4. **Number representation:** no leading zeros, no trailing zeros after decimal, no positive sign, `-0` becomes `0`
5. **String escaping:** minimal escaping -- only control characters, backslash, and double-quote are escaped

```typescript
/**
 * RFC 8785 JSON Canonicalization Scheme (JCS).
 *
 * Produces deterministic JSON output where:
 * - Object keys are sorted lexicographically (Unicode code point order)
 * - No insignificant whitespace
 * - Numbers use shortest representation
 * - Strings use minimal escaping
 *
 * This is critical for IPFS CID reproducibility: any party re-serializing
 * the same data must produce byte-identical output.
 */

/**
 * Canonicalize a value to a deterministic JSON string per RFC 8785.
 *
 * @param value - Any JSON-serializable value
 * @returns Canonical JSON string (no trailing newline)
 */
export function canonicalize(value: unknown): string {
  return serializeValue(value);
}

function serializeValue(value: unknown): string {
  if (value === null) return "null";
  if (value === undefined) return "null"; // treat undefined as null for safety

  switch (typeof value) {
    case "boolean":
      return value ? "true" : "false";
    case "number":
      return serializeNumber(value);
    case "string":
      return serializeString(value);
    case "object":
      if (Array.isArray(value)) {
        return serializeArray(value);
      }
      return serializeObject(value as Record<string, unknown>);
    default:
      throw new Error(`Cannot canonicalize value of type ${typeof value}`);
  }
}

function serializeNumber(num: number): string {
  if (!isFinite(num)) {
    throw new Error("Cannot canonicalize Infinity or NaN");
  }
  // Object.is(-0, num) detects negative zero
  if (Object.is(num, -0)) return "0";
  // JSON.stringify produces the shortest representation for numbers
  return JSON.stringify(num);
}

function serializeString(str: string): string {
  // JSON.stringify handles escaping correctly for RFC 8785
  // It escapes control characters, backslash, and double-quote
  return JSON.stringify(str);
}

function serializeArray(arr: unknown[]): string {
  const items = arr.map((item) => serializeValue(item));
  return "[" + items.join(",") + "]";
}

function serializeObject(obj: Record<string, unknown>): string {
  // Sort keys lexicographically by Unicode code point
  const keys = Object.keys(obj).sort();
  const pairs = keys
    .filter((key) => obj[key] !== undefined) // skip undefined values
    .map((key) => serializeString(key) + ":" + serializeValue(obj[key]));
  return "{" + pairs.join(",") + "}";
}
```

Key decisions:
- Hand-rolled implementation rather than a dependency to minimize attack surface and ensure correctness. The logic is simple and well-specified by RFC 8785.
- `undefined` values in objects are skipped (matching `JSON.stringify` behavior). Top-level `undefined` becomes `"null"`.
- `JSON.stringify` is used for string escaping (it produces RFC 8785-compliant escape sequences).
- Negative zero is normalized to positive zero.
- Infinity and NaN throw errors (they are not valid JSON).

### Task 3: Create Pinata client wrapper

**Files:**
- Create: `src/lib/ipfs/pinata.ts`

- [ ] **Step 1: Create the Pinata client wrapper**

Create `src/lib/ipfs/pinata.ts`:

This module wraps the Pinata SDK and provides three operations:
1. `pinJSON(data, name)` -- pin a JSON object to IPFS, return the CID
2. `pinBlob(blob, name)` -- pin raw bytes to IPFS (for encrypted response data), return the CID
3. `getContent(cid)` -- fetch content from IPFS via the dedicated Pinata gateway

The module lazily initializes the Pinata client on first use and throws a descriptive error if env vars are missing.

```typescript
import { PinataSDK } from "pinata";
import { env } from "~/env";

let pinataClient: PinataSDK | null = null;

/**
 * Get or create the Pinata SDK client.
 * Throws if PINATA_JWT is not configured.
 */
function getClient(): PinataSDK {
  if (pinataClient) return pinataClient;

  if (!env.PINATA_JWT) {
    throw new Error(
      "Pinata credentials not configured. Set PINATA_JWT environment variable (generate from Pinata dashboard).",
    );
  }

  pinataClient = new PinataSDK({
    pinataJwt: env.PINATA_JWT,
    pinataGateway: env.PINATA_GATEWAY_URL ?? undefined,
  });

  return pinataClient;
}

/**
 * Build the gateway URL for fetching content by CID.
 */
function getGatewayUrl(cid: string): string {
  const gateway = env.PINATA_GATEWAY_URL ?? "https://gateway.pinata.cloud";
  // Remove trailing slash if present
  const base = gateway.replace(/\/$/, "");
  return `${base}/ipfs/${cid}`;
}

/**
 * Pin a JSON object to IPFS via Pinata.
 *
 * @param data - The JSON-serializable data to pin (should already be canonicalized)
 * @param name - A human-readable name for the pin (e.g., "survey-0x4d2e...a7f0")
 * @param groupId - Optional Pinata group ID for organizing pins by survey
 * @returns The IPFS CID (Content Identifier) of the pinned data
 */
export async function pinJSON(
  data: Record<string, unknown>,
  name: string,
  groupId?: string,
): Promise<string> {
  const client = getClient();

  let upload = client.upload.json(data).addMetadata({ name });
  if (groupId) {
    upload = upload.group(groupId);
  }

  const result = await upload;
  return result.IpfsHash;
}

/**
 * Pin raw bytes (a Blob) to IPFS via Pinata.
 * Used for encrypted response data (private surveys).
 *
 * @param blob - The data to pin as a File/Blob
 * @param name - A human-readable name for the pin
 * @param groupId - Optional Pinata group ID
 * @returns The IPFS CID of the pinned data
 */
export async function pinBlob(
  blob: Blob,
  name: string,
  groupId?: string,
): Promise<string> {
  const client = getClient();

  const file = new File([blob], name, { type: "application/octet-stream" });
  let upload = client.upload.file(file).addMetadata({ name });
  if (groupId) {
    upload = upload.group(groupId);
  }

  const result = await upload;
  return result.IpfsHash;
}

/**
 * Fetch content from IPFS via the Pinata dedicated gateway.
 *
 * @param cid - The IPFS CID to fetch
 * @returns The response body as a string
 * @throws If the gateway request fails or content is not found
 */
export async function getContent(cid: string): Promise<string> {
  const url = getGatewayUrl(cid);
  const response = await fetch(url);

  if (!response.ok) {
    throw new Error(
      `Failed to fetch IPFS content for CID ${cid}: ${response.status} ${response.statusText}`,
    );
  }

  return response.text();
}

/**
 * Fetch and parse JSON content from IPFS.
 *
 * @param cid - The IPFS CID to fetch
 * @returns The parsed JSON content
 */
export async function getJSON<T = unknown>(cid: string): Promise<T> {
  const content = await getContent(cid);
  return JSON.parse(content) as T;
}

export { getGatewayUrl };
```

Key decisions:
- Lazy client initialization so the module can be imported without env vars being present (for typechecking, tests, etc.).
- `pinJSON` accepts already-structured data. The caller is responsible for canonicalization (done in pin-survey.ts and pin-response.ts).
- `getContent` uses raw `fetch` against the gateway URL rather than the SDK's get method, for simplicity and to avoid SDK-specific quirks.
- The `name` parameter follows the spec's naming convention: `survey-{surveyHash}` or `response-{surveyHash}-{blindedId}`.

**Note on Pinata SDK API:** The exact Pinata SDK API may differ from what's shown above. The implementer should check the actual `pinata` package exports and adjust the client initialization and method calls accordingly. The key contract is: pin JSON -> get CID string, pin file -> get CID string, fetch by CID -> get content.

### Task 4: Create survey and response JSON schemas

**Files:**
- Create: `src/lib/ipfs/schemas.ts`

- [ ] **Step 1: Create the IPFS JSON schemas**

Create `src/lib/ipfs/schemas.ts`:

Define Zod schemas and TypeScript types for the JSON structures pinned to IPFS. These must match the spec exactly (from `docs/superpowers/specs/2026-04-05-ipfs-storage-design.md`). The schemas serve two purposes:
1. **Validation**: ensure data matches the expected format before pinning
2. **Documentation**: make the IPFS content format explicit and type-safe

```typescript
import { z } from "zod";

/**
 * IPFS Survey JSON schema.
 *
 * Pinned at survey publication. Matches the EIP-712 Survey struct exactly
 * so that a verifier can take this JSON, compute the EIP-712 hash, and
 * compare to the on-chain surveyHash.
 *
 * See: docs/superpowers/specs/2026-04-05-ipfs-storage-design.md
 */
export const ipfsSurveyQuestionSchema = z.object({
  text: z.string(),
  questionType: z.string(),
  position: z.number().int().min(0).max(255),
  required: z.boolean(),
  options: z.array(z.string()),
  minRating: z.number().int().min(0).max(255),
  maxRating: z.number().int().min(0).max(255),
  maxLength: z.number().int().min(0).max(65535),
});

export const ipfsSurveySchema = z.object({
  version: z.literal("1"),
  title: z.string(),
  description: z.string(),
  creator: z.string().regex(/^0x[0-9a-fA-F]{40}$/),
  slug: z.string(),
  isPrivate: z.boolean(),
  accessMode: z.enum(["OPEN", "INVITE_ONLY"]),
  resultsVisibility: z.enum(["PUBLIC", "RESPONDENTS", "CREATOR"]),
  questions: z.array(ipfsSurveyQuestionSchema),
});

export type IpfsSurveyJSON = z.infer<typeof ipfsSurveySchema>;

/**
 * IPFS Response Answer schema.
 */
export const ipfsResponseAnswerSchema = z.object({
  questionIndex: z.number().int().min(0).max(255),
  questionType: z.string(),
  value: z.string(),
});

/**
 * IPFS Response JSON schema (public/plaintext survey responses).
 *
 * Pinned at response submission. Contains the blinded respondent ID
 * (not the raw wallet address) and the EIP-712 signature.
 *
 * See: docs/superpowers/specs/2026-04-05-ipfs-storage-design.md
 */
export const ipfsResponseSchema = z.object({
  version: z.literal("1"),
  surveyHash: z.string().regex(/^0x[0-9a-fA-F]{64}$/),
  respondent: z.string().regex(/^0x[0-9a-fA-F]{64}$/), // blinded ID (bytes32)
  answers: z.array(ipfsResponseAnswerSchema),
  signature: z.string().regex(/^0x[0-9a-fA-F]+$/),
});

export type IpfsResponseJSON = z.infer<typeof ipfsResponseSchema>;
```

Key decisions:
- `version` is `z.literal("1")` -- strict check for the current schema version. Future versions will be handled by the verification tools using the version field to select the correct schema.
- **`version` does NOT affect `surveyHash`** -- per the two-layer hashing design (Sub-Plan 2-2): `surveyHash = keccak256(encodeAbiParameters(...))` is computed from the EIP-712 struct fields directly, not from the IPFS JSON bytes. The IPFS JSON is a separate human/tool-readable artifact. A verifier recomputes the hash by extracting the struct fields from the IPFS JSON and passing them to `hashSurvey()` -- the `version` field is metadata that is ignored by the hash function. This means the IPFS schema can evolve (add fields, bump version) without invalidating existing hashes.
- `creator` is validated as a 20-byte hex address.
- `surveyHash` and `respondent` (blinded ID) are validated as 32-byte hex strings.
- `signature` is a hex string of variable length (65 bytes = 130 hex chars for ECDSA, but we don't enforce exact length for future-proofing).
- Numeric fields use int constraints matching the EIP-712 types (uint8 = 0-255, uint16 = 0-65535).

### Task 5: Create survey pinning function

**Files:**
- Create: `src/lib/ipfs/pin-survey.ts`

- [ ] **Step 1: Create the survey pinning function**

Create `src/lib/ipfs/pin-survey.ts`:

This function is called by the PUBLISH_SURVEY job handler. It:
1. Builds the IPFS survey JSON from the database survey record
2. Validates it against the schema
3. Serializes to deterministic JSON (RFC 8785)
4. Pins to Pinata
5. Returns the CID

```typescript
import { canonicalize } from "./deterministic-json";
import { ipfsSurveySchema, type IpfsSurveyJSON } from "./schemas";
import { pinBlob } from "./pinata";

/**
 * Input data for pinning a survey to IPFS.
 * This is assembled from the database Survey + Question records
 * by the PUBLISH_SURVEY job handler.
 */
export interface PinSurveyInput {
  title: string;
  description: string;
  creator: string; // wallet address (0x...)
  slug: string;
  isPrivate: boolean;
  accessMode: "OPEN" | "INVITE_ONLY";
  resultsVisibility: "PUBLIC" | "RESPONDENTS" | "CREATOR";
  questions: Array<{
    text: string;
    questionType: string;
    position: number;
    required: boolean;
    options: string[];
    minRating: number;
    maxRating: number;
    maxLength: number;
  }>;
}

/**
 * Pin a survey to IPFS via Pinata.
 *
 * 1. Build the IPFS survey JSON (with version field)
 * 2. Validate against the schema
 * 3. Serialize to deterministic JSON (RFC 8785)
 * 4. Pin to Pinata with naming convention "survey-{surveyHash}"
 * 5. Return the CID
 *
 * @param input - Survey data assembled from the database
 * @param surveyHash - The EIP-712 hash of the survey (used for pin naming)
 * @returns The IPFS CID of the pinned survey JSON
 */
export async function pinSurvey(
  input: PinSurveyInput,
  surveyHash: string,
): Promise<string> {
  // Build the IPFS JSON structure
  const surveyJSON: IpfsSurveyJSON = {
    version: "1",
    title: input.title,
    description: input.description,
    creator: input.creator,
    slug: input.slug,
    isPrivate: input.isPrivate,
    accessMode: input.accessMode,
    resultsVisibility: input.resultsVisibility,
    questions: input.questions.map((q) => ({
      text: q.text,
      questionType: q.questionType,
      position: q.position,
      required: q.required,
      options: q.options,
      minRating: q.minRating,
      maxRating: q.maxRating,
      maxLength: q.maxLength,
    })),
  };

  // Validate against schema (throws ZodError if invalid)
  ipfsSurveySchema.parse(surveyJSON);

  // Serialize to deterministic JSON bytes (RFC 8785).
  // We pin the raw canonical bytes (not a parsed object) so that Pinata stores
  // exactly the bytes we produce. This ensures the CID is reproducible:
  // any verifier who canonicalizes the same JSON will get the same bytes
  // and therefore the same CID.
  const canonical = canonicalize(surveyJSON);
  const canonicalBytes = new TextEncoder().encode(canonical);
  const blob = new Blob([canonicalBytes], { type: "application/json" });

  const pinName = `survey-${surveyHash.slice(0, 10)}`;
  const cid = await pinBlob(blob, pinName);

  return cid;
}

/**
 * Build an IpfsSurveyJSON object from input data.
 * Useful for verification (re-building the JSON to compare hashes)
 * without actually pinning to IPFS.
 */
export function buildSurveyJSON(input: PinSurveyInput): IpfsSurveyJSON {
  return {
    version: "1",
    title: input.title,
    description: input.description,
    creator: input.creator,
    slug: input.slug,
    isPrivate: input.isPrivate,
    accessMode: input.accessMode,
    resultsVisibility: input.resultsVisibility,
    questions: input.questions.map((q) => ({
      text: q.text,
      questionType: q.questionType,
      position: q.position,
      required: q.required,
      options: q.options,
      minRating: q.minRating,
      maxRating: q.maxRating,
      maxLength: q.maxLength,
    })),
  };
}
```

Key decisions:
- Schema validation happens before pinning to catch data issues early.
- `buildSurveyJSON` is exported separately for use in verification flows (where you need to rebuild the JSON without pinning).
- Pin name uses a truncated hash for readability in the Pinata dashboard.
- The canonical JSON is parsed back to an object for Pinata because the SDK expects objects. This is a known limitation -- the actual CID depends on Pinata's internal serialization. For strict CID verification, the open-source tools will canonicalize and compute CIDs locally.

### Task 6: Create response pinning function

**Files:**

- [ ] **Step 1: Create the response pinning function**

Create `src/lib/ipfs/pin-response.ts`:

This function is called by the SUBMIT_RESPONSE job handler. It:
1. Builds the IPFS response JSON from the database response record
2. For public surveys: validates and pins plaintext JSON
3. For private surveys: encrypts the JSON and pins the encrypted blob (encryption implementation deferred to Phase 3 private surveys spec, but the pinning interface supports it now)
4. Returns the CID

```typescript
import { canonicalize } from "./deterministic-json";
import { ipfsResponseSchema, type IpfsResponseJSON } from "./schemas";
import { pinJSON, pinBlob } from "./pinata";

/**
 * Input data for pinning a response to IPFS.
 * Assembled from the database Response + Answer records
 * by the SUBMIT_RESPONSE job handler.
 */
export interface PinResponseInput {
  surveyHash: string; // bytes32 hex
  blindedId: string; // bytes32 hex (blinded respondent ID)
  answers: Array<{
    questionIndex: number;
    questionType: string;
    value: string;
  }>;
  signature: string; // EIP-712 signature hex
}

/**
 * Pin a response to IPFS via Pinata (plaintext, for public surveys).
 *
 * 1. Build the IPFS response JSON (with version field)
 * 2. Validate against the schema
 * 3. Serialize to deterministic JSON (RFC 8785)
 * 4. Pin to Pinata with naming convention "response-{surveyHash}-{blindedId}"
 * 5. Return the CID
 *
 * @param input - Response data assembled from the database
 * @returns The IPFS CID of the pinned response JSON
 */
export async function pinResponse(input: PinResponseInput): Promise<string> {
  // Build the IPFS JSON structure
  const responseJSON: IpfsResponseJSON = {
    version: "1",
    surveyHash: input.surveyHash,
    respondent: input.blindedId,
    answers: input.answers.map((a) => ({
      questionIndex: a.questionIndex,
      questionType: a.questionType,
      value: a.value,
    })),
    signature: input.signature,
  };

  // Validate against schema (throws ZodError if invalid)
  ipfsResponseSchema.parse(responseJSON);

  // Serialize to deterministic JSON
  const canonical = canonicalize(responseJSON);

  // Pin to Pinata
  const pinName = `response-${input.surveyHash.slice(0, 10)}-${input.blindedId.slice(0, 10)}`;
  const cid = await pinJSON(
    JSON.parse(canonical) as Record<string, unknown>,
    pinName,
  );

  return cid;
}

/**
 * Pin an encrypted response to IPFS via Pinata (for private surveys).
 *
 * The encrypted blob is an AES-256-GCM ciphertext of the canonical
 * response JSON. Encryption is handled by the caller (the encryption
 * module will be implemented in a future private surveys plan).
 *
 * @param encryptedData - The encrypted response data as a Uint8Array
 * @param surveyHash - The survey hash (for pin naming)
 * @param blindedId - The blinded respondent ID (for pin naming)
 * @returns The IPFS CID of the pinned encrypted data
 */
export async function pinEncryptedResponse(
  encryptedData: Uint8Array,
  surveyHash: string,
  blindedId: string,
): Promise<string> {
  const pinName = `response-${surveyHash.slice(0, 10)}-${blindedId.slice(0, 10)}-encrypted`;
  const blob = new Blob([encryptedData], { type: "application/octet-stream" });
  const cid = await pinBlob(blob, pinName);
  return cid;
}

/**
 * Build an IpfsResponseJSON object from input data.
 * Useful for verification without actually pinning.
 */
export function buildResponseJSON(input: PinResponseInput): IpfsResponseJSON {
  return {
    version: "1",
    surveyHash: input.surveyHash,
    respondent: input.blindedId,
    answers: input.answers.map((a) => ({
      questionIndex: a.questionIndex,
      questionType: a.questionType,
      value: a.value,
    })),
    signature: input.signature,
  };
}
```

Key decisions:
- `pinResponse` handles public (plaintext) survey responses.

import { describe, it, expect } from "vitest";
import { canonicalize } from "../deterministic-json";

describe("canonicalize (RFC 8785 JCS)", () => {
  it("sorts object keys lexicographically", () => {
    const result = canonicalize({ z: 1, a: 2, m: 3 });
    expect(result).toBe('{"a":2,"m":3,"z":1}');
  });

  it("sorts nested object keys", () => {
    const result = canonicalize({ b: { z: 1, a: 2 }, a: 1 });
    expect(result).toBe('{"a":1,"b":{"a":2,"z":1}}');
  });

  it("produces no whitespace", () => {
    const result = canonicalize({ key: "value", arr: [1, 2, 3] });
    expect(result).not.toContain(" ");
    expect(result).not.toContain("\n");
    expect(result).not.toContain("\t");
  });

  it("handles arrays", () => {
    const result = canonicalize([3, 1, 2]);
    expect(result).toBe("[3,1,2]"); // arrays preserve order
  });

  it("handles nested arrays", () => {
    const result = canonicalize({ items: [{ b: 2, a: 1 }] });
    expect(result).toBe('{"items":[{"a":1,"b":2}]}');
  });

  it("handles null", () => {
    expect(canonicalize(null)).toBe("null");
  });

  it("handles booleans", () => {
    expect(canonicalize(true)).toBe("true");
    expect(canonicalize(false)).toBe("false");
  });

  it("handles strings with special characters", () => {
    const result = canonicalize({ msg: 'Hello "world"\nNew line' });
    expect(result).toBe('{"msg":"Hello \\"world\\"\\nNew line"}');
  });

  it("handles empty objects and arrays", () => {
    expect(canonicalize({})).toBe("{}");
    expect(canonicalize([])).toBe("[]");
  });

  it("normalizes negative zero to zero", () => {
    expect(canonicalize(-0)).toBe("0");
  });

  it("handles integers", () => {
    expect(canonicalize(42)).toBe("42");
    expect(canonicalize(-17)).toBe("-17");
  });

  it("handles floating point numbers", () => {
    expect(canonicalize(3.14)).toBe("3.14");
  });

  it("throws on Infinity", () => {
    expect(() => canonicalize(Infinity)).toThrow();
  });

  it("throws on NaN", () => {
    expect(() => canonicalize(NaN)).toThrow();
  });

  it("skips undefined object values", () => {
    const result = canonicalize({ a: 1, b: undefined, c: 3 });
    expect(result).toBe('{"a":1,"c":3}');
  });

  it("is deterministic (same input always produces same output)", () => {

      title: "Test Survey",

    expect(result1).toBe(result2);
